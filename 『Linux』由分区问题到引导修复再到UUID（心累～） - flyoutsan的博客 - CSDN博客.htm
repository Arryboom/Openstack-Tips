『Linux』由分区问题到引导修复再到UUID（心累～）

标签： linux引导修复grub rescue多系统
2016-07-20 19:17 1930人阅读 评论(0) 收藏 举报
分类：

版权声明：本文为博主原创文章，未经博主允许不得转载。

问题篇： 

一切问题都从强迫症“移动”了fedora / 分区起源。”移动“的目的是将不连续的空闲空间利用起来，增加windows的分区容量（之前为了玩linux把windows的空间剥削的太严重了，现 在空间略显不足==）。为什么说是“移动”呢？因为实际上是删除重装，介于先前一次”移动失败“的惨痛教训，最终决定”稳一点“，索性直接重装。于是又是 熟悉的流程： 制作启动盘 —— 分区 —— 安装， 事情应该十分顺利。 然而由于是”移动“， 所以试图省事在分区的步骤中就直接删除原来fedora除了/home的所有分区（/swap和/boot/etc），/home作为新系统的/home 分区，新划分一个约为2G的/swap分区（原来分区是8G，发现4G内存根本不需要这么多），同时经提示划分一个新的/boot/efi分区，十分流畅 的完成安装。安装过后发现， 除了新的fedora其他系统引导全部丢失。电脑中算上fedora、windows一共由三个系统，还有一个是kali。然后不知为何fedora在安 装时就检测不到磁盘中windows的存在，在启动后fedora的启动项中就没有windows，而kali也显示的命令不存在。这下问题大了，原来的 三个系统全部阵亡。

分析篇：

怎么破？二话不说上PE。打开PE中disk genius，我一看，分区中貌似少了点什么，最前面的ESP哪去了？为什么是空的？肯定是被谁给删了 —— 没错，那个“谁”就是我自己==。之前顺利得不得了的操作中，有一步作了这个死，那就是“删除原来fedora除了/home的所有分区（/，/swap 和/boot/efi）“。原来这里fedora的/boot/efi就是 /ESP分区！然而fedora安装程序的的分区列表中直接把/ESP分区写作了/boot/efi分区并且列在了fedora系统的子分区中，而没有意 识到这个问题的我傻乎乎的把它给删了于是乎引导全部丢失。废话了这么多实际上问题就是：不慎删除了/ESP分区。另外，由于新建了分区导致分区也有所变 化。

解决篇：

由于先前没有备份/ESP分区，所以还原是没戏了。那么试一试数据恢复如何？恰好disk genius就有这个功能，不过选项是暗的。因为分区都给删了，233。于是用disk genius先新建一个/ESP分区，格式化（不格式化也检测不到），将找到的文件全部复制到别的磁盘，再复制回来。看一下恢复的目录，和原来的差不多， 除了部分字母变成了！，还多出了一个“丢失的文件”目录，把！通过比较推理改回来，重新引导试试。很遗憾，windows报错，建议插入修复光盘，而 kali则是进入了grub rescue。无奈，恢复的路不好走。只能走添加引导的路了，还是PE，使用windows引导修复工具，找到新建立的/ESP的盘符和windows的 安装盘盘符，添加引导，失败。为何会失败？这不科学。是不是因为恢复的文件的关系？把ESP的再格式化一遍，再试试看，果然，这次成功了。重启看看效果， 终于看到熟悉的windows引导画面，进去后一切仿佛依旧（毕竟只是引导出了问题，系统文件、程序、数据什么的还是正常的）。搞定了windows，再 来搞定kali。首先试试grub rescue手动引导，步骤是ls找到所有的分区，逐个ls (hd*,gpt#)/boot，直到没有报错的为止, 由于只有一块硬盘 ，这里的*就是0，#是所有分区的编号，譬如说我的有16个分区，就是1 - 16，从前往后递增编号。最终试出(hd0,gpt11)中没有报错，接下来（http://askubuntu.com/questions/493826/grub-rescue-problem-after-deleting-ubuntu-partition)

set root=(hd0,gpt11)

set prefix=(hd0,gpt11)/boot/grub #这里是grub的目录，有的可能不一样

insmod normal

normal

接着就出现了熟悉的kali引导器引导画面了，三个启动项都有，似乎胜利在望。然而，这也只是“似乎”。新的风暴已经出现，kali无法进入系统， 似乎是加载某个设备失败，倒计时了1分30秒就进入了emergency模式，这个模式是纯命令模式，可以执行正常模式的一切非图形操作。我们可以通过 journalctl -xb查看日志信息，错误用红色的标出来了，错误大概是分区变动导致原来的对应关系不可用，譬如说/home原来在/dev/sda13现在到了/dev /sda12，需要我们手动改回来。这个对应关系存储在/etc/fstab中，磁盘使用的是UUID来标记，使用命令blkid可以查看所有分区的 UUID，将其改回正确即可。到了这里，进入kali已经成功解决了。然而还没完，还记得之前是手动引导的吗？我们还需要安装kali引导。安装可以用启 动盘也可以手动引导进系统了再装。事实上我是先用启动盘修复的引导再改/etc/fstab文件，用启动盘的修复的过程理论上为：

mount /dev/sda11 / mnt/

mount --bind /dev /mnt/dev

mount --bind /proc /mnt/proc

mount --bind /sys /mnt/sys

chroot /mnt

grub-install /dev/sda

update-grub

exit

umount /mnt/dev

umount /mnt/proc

umount /mnt/sys

umount /mnt

然而这对我的电脑似乎不行，因为报错：cant't find EFI dictionary，可能是UEFI引导模式的关系,还需要挂载efi分区，最终找到解决方案http://askubuntu.com/questions/53578/can-i-install-in-uefi-mode-with-the-alternate-installer/57380#57380，解决如下：

### Mounting ###


sudo mount /dev/sda# /mnt            #Mount root (/) partition
sudo mount /dev/sda# /mnt/boot       #Mount boot (/boot) partition 
                                      (if separate from root partition)
sudo mkdir -p /mnt/boot/efi          #Create EFI partition mount point
sudo mount /dev/sda1 /mnt/boot/efi   #Mount EFI partition


sudo mount --bind /dev /mnt/dev
sudo mount --bind /proc /mnt/proc
sudo mount --bind /sys /mnt/sys


sudo chroot /mnt                     #Chroot to your installation


### Installing ###


apt-get install grub-efi-amd64  #Install grub EFI bootloader


grub-install --recheck --no-floppy --force
                                     #Install grub bootloader in EFI partition


echo "configfile (hd0,gpt#)/boot/grub.cfg" > /boot/efi/ubuntu/grub.cfg
                                     #Tell grub to load grub.cfg from /boot


update-grub                          #Create grub menu list


exit                                 #Exit chroot


### Unmounting ###


sudo umount /mnt/dev
sudo umount /mnt/proc
sudo umount /mnt/sys
sudo umount /mnt/boot/efi
sudo umount /mnt/boot
sudo umount /mnt

总结：

没事少动分区，备份十分重要。

下手前请三思，数据安全第一。
